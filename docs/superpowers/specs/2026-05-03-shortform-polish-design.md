# 短片轻量包装增强设计

## 背景

当前 ClipForge Agent 已经具备以下能力：

1. 生成剪辑方案
2. 搜索并下载素材
3. 为每段素材计算裁剪窗口
4. 输出接近 30 秒的竖屏 `9:16` 成片

这一阶段已经把系统从“能跑通”推进到了“能产出结构正确的短视频”。但对用户来说，当前成片和工作台仍然偏工程态：

- 成片没有字幕
- 成片没有统一背景音乐
- 前端只能看到粗粒度状态
- 用户不容易直观看到每段素材实际是怎么被裁剪和使用的

本设计聚焦于在不显著增加系统复杂度的前提下，为短片增加一层轻量包装，并同步增强前端的过程和结果可见性。

## 目标

本阶段交付四项增强能力：

1. 每段自动字幕
2. 一条通用背景音乐
3. 更细粒度的 Agent 进度展示
4. 结果面板中的片段清单与裁剪信息展示

## 非目标

本阶段明确不做：

- 字幕编辑器
- 多套 BGM 风格库
- 自动润色字幕文案
- 封面图或封面帧生成
- 复杂转场动效
- 复杂音频 ducking
- 前端参数调节面板

## 方案选择

### 方案 A：轻量模板增强

- 字幕文案直接使用 `plan.scenes[].description`
- 每段 clip 全时长显示对应字幕
- 使用固定通用 BGM
- 前端展示裁剪结果和字幕信息

优点：

- 改动集中
- 稳定性高
- 非常适合在现有链路上快速提升体验

缺点：

- 字幕文案偏结构化，不像成品营销文案
- BGM 风格固定

### 方案 B：半智能包装增强

- 在 A 的基础上生成更自然的字幕短句
- 根据 style 选择 BGM

优点：

- 更接近成品视频

缺点：

- 增加新的文案和风格选择链路
- 容易把本阶段变成另一个 agent 子系统

### 方案 C：完整包装管线

- 片头、片尾、字幕、封面、BGM 一次做完

优点：

- 展示效果最好

缺点：

- 超出当前单阶段合理范围

### 结论

采用 **方案 A：轻量模板增强**。

## 字幕设计

### 字幕来源

每段字幕直接使用该场景的：

- `scene.description`

例如：

- 开场建立氛围
- 展示核心功能或主题
- 呈现真实使用场景

### 字幕时序

- 每段 clip 的字幕覆盖该 clip 的完整时长
- 不引入额外时间轴编辑系统
- 字幕与裁剪后的片段严格一一对应

### 字幕样式

采用通用安全风：

- 白色主文字
- 半透明深色底或描边
- 底部居中显示
- 字号中等偏大，以移动端可读性优先

样式目标：

- 可读
- 不抢画面主体
- 不强烈偏向平台娱乐风或品牌广告风

## 背景音乐设计

### 音乐来源

使用仓库内置的一条默认背景音乐：

- 路径建议：`backend/assets/audio/default_bgm.mp3`

### 使用规则

- 全片统一铺底
- 若 BGM 时长不足，则循环补齐
- 若 BGM 时长超出成片时长，则裁切到成片结束

### 音量策略

- BGM 音量默认压低
- 原视频音轨保留
- 不做复杂自动压混，仅做轻量混音

这一策略的目标不是广播级混音，而是在保持稳定的前提下增加成片完成度。

## Agent 事件与结果增强

### clip metadata 扩展

在现有 clip artifact metadata 中增加：

- `caption`
- `sourceDuration`
- `trimStart`
- `trimDuration`

这样前端无需推导，直接可视化展示 Agent 的决策结果。

### 事件增强

在渲染阶段增加更细的事件提示，例如：

- 正在裁剪片段
- 正在合成字幕
- 正在混合背景音乐
- 正在输出最终视频

不新增状态枚举，只在现有 `rendering` 阶段提供更细消息。

## 前端展示增强

### ProgressPanel

增强目标：

- 继续展示总体进度
- 在 `rendering` 阶段显示更细粒度文案
- 最近事件比当前更有解释性

例如用户能看到：

- 正在裁剪片段
- 正在叠加字幕
- 正在混合背景音乐

### ResultPanel

增强目标：

- 保留视频预览与下载
- 增加“片段清单”

每段片段展示：

- 场景字幕文案
- 目标时长
- 原始素材时长
- 裁剪起点
- 裁剪时长
- 素材来源链接或标识

结果页要让用户明确看到：

- Agent 找到了哪些素材
- 每段实际剪了哪一段
- 每段字幕是什么

## 模块职责

### `backend/services/render_service.py`

负责：

- 给裁剪后片段叠加字幕
- 生成全片 BGM 音频流
- 混合原视频音频与背景音乐
- 输出最终短片

### `backend/tasks/agent_tasks.py`

负责：

- 在持久化 clip artifact 时写入 `caption`
- 在渲染阶段记录更细事件

### `backend/services/agent_read_service.py`

负责：

- 从 artifact metadata 中读回字幕和裁剪信息
- 返回给前端可直接展示的 clip 数据

### `src/components/agent/ProgressPanel.tsx`

负责：

- 展示更细进度信息
- 展示最近关键事件

### `src/components/agent/ResultPanel.tsx`

负责：

- 展示最终视频
- 展示每段 clip 的结构化清单

## 测试策略

### 后端测试

至少覆盖：

1. clip metadata 包含 `caption`
2. 渲染配置包含字幕和 BGM 信息
3. 任务成功链路在增加包装能力后仍然可用

### 前端验证

至少验证：

1. `ProgressPanel` 能显示更细的渲染阶段事件
2. `ResultPanel` 能展示 clip 清单和字幕文案

### 真实联调

执行一次真实任务，确认：

1. 视频仍可正常产出
2. 成片可见字幕
3. 成片存在背景音乐
4. 前端能展示裁剪信息与字幕文案

## 风险与折中

### 风险 1：字幕遮挡主体

当前使用固定底部字幕布局，某些镜头可能遮挡画面主体。

接受该折中，后续再考虑智能避让。

### 风险 2：BGM 与原视频音轨冲突

当前只做轻量混音，不做复杂压混，因此少数素材可能出现声音层级不理想。

接受该折中，优先保证实现成本和稳定性。

### 风险 3：字幕文案不够“像文案”

直接使用 `scene.description` 会偏结构化。

接受该折中，因为本阶段目标是增强完成度，而不是新建文案系统。

## 验收标准

满足以下条件即可视为本阶段完成：

1. 成片有字幕
2. 成片有背景音乐
3. 现有竖屏 30 秒左右输出能力不被破坏
4. 前端能展示更细的渲染进度
5. 前端结果区能看到每段素材的字幕和裁剪信息
