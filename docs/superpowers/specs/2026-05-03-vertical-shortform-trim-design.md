# 竖屏智能短片裁剪设计

## 背景

当前 Agent 工作流已经可以完成以下链路：

1. 生成剪辑方案
2. 搜索并下载素材
3. 触发 Celery worker 执行
4. 渲染并输出最终视频

但现有渲染逻辑仍然是“把下载到的整段素材直接拼接”，没有真正使用 `plan.scenes[].duration`。这会导致最终产物虽然流程跑通，但时长和节奏完全失控，例如目标 30 秒的任务实际导出了 12 分钟以上的视频。

本设计聚焦于把当前系统从“能跑通”推进到“能产出结构正确的短视频”。

## 目标

本阶段只实现基础可用版：

- 输出固定为竖屏 `9:16`
- 总时长尽量贴近 `plan.targetDuration`
- 每个场景按 `plan.scenes[].duration` 真实裁剪
- 保持现有 Agent 工作流和持久化链路不变
- 不引入字幕、BGM、复杂转场或内容理解模型

## 非目标

本阶段明确不做以下内容：

- 多候选素材评分与排序优化
- 人脸识别、主体跟随、精彩片段检测
- 自动字幕、背景音乐、封面帧
- 前端交互重构
- 整体任务链路重构

## 方案选择

### 方案 A：固定时长裁剪 + 统一竖屏导出

每个场景下载一段候选素材，按场景目标时长裁剪，再统一缩放裁切为竖屏并顺序拼接。

优点：

- 改动集中，风险低
- 能直接解决“输出时长失控”的核心问题
- 与现有 Agent 计划模型天然兼容

缺点：

- 不具备素材语义理解
- 当前镜头截取规则较朴素

### 方案 B：多候选打分后裁剪

每个场景保留多个候选素材，按简单规则打分后选最优素材再裁剪。

优点：

- 可提高素材命中率

缺点：

- 会扩大链路复杂度
- 当前阶段收益不如方案 A 明显

### 方案 C：先整段拼接，再二次压缩总时长

优点：

- 实现快

缺点：

- 破坏场景节奏
- 不能真正体现 Agent 的剪辑意图

### 结论

采用 **方案 A**。

## 数据模型调整

增强 `backend.models.agent.ClipInfo`，增加最小必要剪辑元数据：

- `sourceDuration`: 原始素材时长
- `trimStart`: 实际裁剪起点，单位秒
- `trimDuration`: 实际裁剪时长，单位秒

保留现有字段：

- `sceneId`
- `sourceUrl`
- `localPath`
- `publicUrl`
- `duration`

其中：

- `duration` 继续表达“该场景计划时长”
- `trimDuration` 表达“本次渲染实际要截出的时长”

这样后续即使扩展更复杂的剪辑策略，也不需要推翻当前结构。

## 裁剪规则

### 基本规则

每个场景只输出一个目标片段。

### 时长规则

- 目标时长取自 `scene.duration`
- 如果原素材时长小于或等于目标时长：
  - `trimStart = 0`
  - `trimDuration = sourceDuration`
- 如果原素材时长大于目标时长：
  - 从中段偏前位置开始裁剪
  - 具体默认规则：
    - `available = sourceDuration - targetDuration`
    - `trimStart = max(0, available * 0.35)`
    - `trimDuration = targetDuration`

选择“中段偏前”的原因：

- 能避开部分片头黑场、开头提示语
- 也能减少片尾 Logo、订阅提醒等尾部噪音
- 规则简单，可预测，便于测试

### 画面规则

统一导出为竖屏 `720x1280`：

1. 缩放到覆盖目标画布
2. 居中裁切到 `720x1280`
3. 统一帧率到 `30fps`
4. 统一编码为 `h264 + aac`

选择 `720x1280` 而不是 `1080x1920` 的原因：

- 更适合当前本地联调速度
- 降低 FFmpeg 渲染压力
- 作为基础可用版已经足够

## 模块职责

### `backend/services/search_service.py`

负责：

- 搜索素材
- 下载素材
- 读取素材原始时长
- 计算默认 `trimStart / trimDuration`
- 返回包含剪辑元数据的 `ClipInfo`

不负责：

- 视频标准化
- 最终合成

### `backend/services/render_service.py`

负责：

- 根据 `ClipInfo.trimStart / trimDuration` 对每段素材裁剪
- 对每段片段统一 scale / crop / fps / 编码参数
- 顺序拼接为最终短片

不再允许直接拼接原始整段素材。

### `backend/tasks/agent_tasks.py`

维持现有工作流：

- 读取计划
- 执行搜索下载
- 持久化素材记录
- 执行渲染
- 持久化最终视频

本阶段不改变该任务编排结构。

## 渲染流程

建议渲染流程如下：

1. 接收 `ClipInfo[]`
2. 校验每段素材文件存在
3. 对每段素材执行：
   - `trim`
   - `scale`
   - `crop`
   - `fps`
   - 音频标准化
4. 生成标准化中间片段
5. 将中间片段按顺序 concat
6. 输出最终 mp4 文件

为提升兼容性，不直接依赖输入文件的编码一致性，而是先统一中间片段格式，再进行合成。

## 错误处理

### 搜索阶段

- 搜不到素材：跳过该场景
- 候选素材下载失败：继续尝试下一个候选
- 所有候选失败：记录日志并跳过该场景

### 渲染阶段

- 输入文件缺失：直接失败
- FFmpeg 裁剪失败：直接失败
- 最终无可用片段：抛出明确错误，例如“没有可渲染的有效片段”

### 状态语义

本阶段不新增 session/job 状态枚举，只沿用现有：

- `searching`
- `downloading`
- `rendering`
- `done`
- `failed`

## 测试策略

至少覆盖以下测试：

1. `ClipInfo` 支持新的剪辑字段
2. 搜索下载阶段会给素材生成 `sourceDuration / trimStart / trimDuration`
3. 渲染阶段会使用 `trimDuration`，而不是整段原视频直接拼接
4. 任务执行成功后，最终产物路径和会话状态仍然正确

测试优先级：

- 先补单元测试和任务级回归测试
- 再做一次真实会话联调，验证导出视频时长接近 30 秒

## 风险与折中

### 风险 1：素材过短

如果候选素材短于目标时长，当前策略会保留整段，可能导致最终总时长略短于计划值。

接受该折中，优先保证流程稳定。

### 风险 2：居中裁切可能切掉主体

当前不做主体检测，因此竖屏裁切存在误伤主体的可能。

接受该折中，后续可通过智能构图升级。

### 风险 3：中间片段渲染增加耗时

相比直接拼接，标准化中间片段会增加一次处理成本。

接受该折中，因为它换来了更稳定的最终合成结果。

## 验收标准

满足以下条件即可视为本阶段完成：

1. 新任务生成的视频比例为竖屏 `9:16`
2. 总时长接近 `plan.targetDuration`
3. 每段场景时长明显受 `scene.duration` 控制
4. 不再出现把多个完整长视频直接拼成超长成片的情况
5. 现有 Agent 入队、事件记录、产物落库链路保持可用
