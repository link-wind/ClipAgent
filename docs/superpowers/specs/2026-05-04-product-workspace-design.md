# ClipForge Product Workspace Design

## Background

ClipForge 当前首页仍然是偏开发工作台式的 Agent 界面：对话、计划、进度、结果并列展示，能工作，但整体更像内部工具，不像可以长期使用的产品。

这次设计调整不改后端能力边界，重点是把前端重构成一个更明确的产品工作空间，让用户在进入系统后能清楚感知三个核心场景：

1. 先看整体数据和项目状态。
2. 再在单栏方案沟通页里完成需求理解、方向选择和方案确认。
3. 最后在任务管理页里统一查看执行状态、处理异常和追踪结果。

本轮设计已经通过静态页面反复确认，后续实现应以这些确认结论为准，而不是重新发散页面方向。

## Goals

1. 让 ClipForge 首页更像产品控制台，而不是开发调试面板。
2. 把用户主流程拆成三个清晰页面：数据总览、方案沟通、任务管理。
3. 保留现有 Agent 工作流能力，但重组信息层级与交互呈现。
4. 优先提升可扫读性、状态透明度和任务可管理性。
5. 后续实现尽量复用当前 Next.js、React、Zustand、CSS Modules 结构。

## Non-Goals

- 本轮不新增后端业务能力。
- 本轮不设计营销首页。
- 本轮不引入复杂素材编辑器、时间轴或拖拽编排。
- 本轮不扩展新的项目协作权限体系。
- 静态稿不直接作为生产代码使用，只作为实现参考。

## Chosen Direction

采用已经确认的 **C 方向 / Product Workspace** 风格。

这套方向的特征是：

- 产品感优先，避免“实验台”“控制台命令面板”的观感。
- 布局克制、密度适中、信息清楚，适合持续使用。
- 色彩偏石墨灰、柔和绿色、低饱和蓝绿，避免过亮荧光色。
- 页面重点是状态、内容与操作，而不是装饰性视觉。

相较其他方向，这个方案最适合 ClipForge 从“能跑”走向“可用、可信、可管理”的产品形态。

## Information Architecture

本轮前端重构以三个一级页面为核心：

1. `/` 或首页仪表盘：数据展示屏。
2. `/workspace` 或具体会话页：单栏方案沟通页面。
3. `/tasks`：任务管理页面。

建议保留当前 Agent 会话能力作为方案沟通页的核心数据来源，并在不打断后端接口的前提下，逐步把现有首页职责拆分到这三个页面中。

## Design Language

### Tone

- 专业、安静、稳定。
- 像产品工作台，不像开发者调试面板。
- 在“可信”和“效率”之间取平衡，不走花哨路线。

### Palette

- 背景：浅灰绿白或浅暖灰，保证长时间阅读舒适。
- 主色：深石墨灰，用于导航、主按钮、重点文字。
- 辅助强调：柔和绿色，用于选中、成功、确认状态。
- 数据色：低饱和蓝绿、灰绿、浅橄榄等组合，避免高亮荧光绿。
- 错误色：保留清晰红色语义，但控制饱和度。

### UI Constraints

- 卡片和弹窗圆角保持 8px 或以下。
- 页面区块采用完整带状布局或平面容器，不做大面积悬浮卡片堆叠。
- 图标优先承担操作识别，按钮文案保持短促。
- 字号层级稳定，不随视口宽度缩放。
- 所有固定格式区域需要有稳定尺寸，避免内容跳动。

## Page 1: Dashboard

## Purpose

首页是数据展示屏，不承担详细沟通，也不承担任务深度编辑。它的职责是帮助用户快速理解：

- 当前项目和任务整体情况。
- 内容生产效率和完成趋势。
- 最近活跃任务与资产状态。
- 从哪里进入新的沟通或管理动作。

## Layout

桌面端采用左侧窄导航加右侧主内容区结构：

- 左侧为产品导航 rail。
- 顶部为标题区、搜索区和关键操作。
- 主体采用 “主内容区 + 侧边状态区” 双区布局。

主内容区建议包含：

1. 顶部关键指标卡。
2. 内容趋势或任务趋势图表。
3. 资产构成或项目结构概览。
4. 最近项目 / 最近产出列表。

侧边状态区建议包含：

1. 最近任务。
2. 待处理事项。
3. 快速入口，例如进入方案沟通页或任务页。

## Content Rules

- 不再出现“快速创建”这类偏空泛、偏工具启动器的模块。
- 首页模块都应围绕“项目状态、数据表现、任务进度、资产情况”组织。
- 图表配色使用克制的数据色，不使用高亮荧光绿。
- 所有指标卡都要能一眼看懂含义，不用长段解释。

## Interaction

- 点击最近项目或最近任务，可进入对应会话页或任务详情。
- 顶部主按钮用于进入新方案沟通，而不是直接“快速创建”。
- 搜索优先服务于任务、项目、会话定位。

## Page 2: Brief Conversation Workspace

## Purpose

第二页是核心业务页面，承担“需求进入系统后如何被理解、拆解、比较、确认”的全过程。

这里不是多栏编辑器，也不是并排工具面板，而是一个单栏、顺序推进的方案沟通界面，让用户像在和一个产品化 Agent 协作，而不是在操作后台配置项。

## Confirmed Layout

必须采用 **单栏结构**。

页面从上到下依次为：

1. 顶部标题与当前会话状态。
2. 对话线程主体。
3. AI 分析步骤流。
4. 方案选项区。
5. 最终确认方案区。
6. 底部输入区。

不使用三栏方案，不保留持续常驻的右侧计划栏。

## Conversation Model

### User Message

用户输入应保持原始表达，不应在用户气泡中混入目标、格式、时长等结构化分析字段。

例如：

- 用户说“我想做一支 30 秒的 AI 笔记产品宣传片……”
- 那么用户气泡只显示这段原始需求。
- “目标”“格式”“使用场景”“风格判断”等内容应作为 AI 的分析结果出现，而不是改写进用户消息。

### AI Processing Flow

AI 每个步骤必须先展示进行中的状态，再展示结果。

每个步骤块都遵循同一结构：

1. 步骤标题。
2. 当前状态或进度条。
3. 步骤完成后的结果内容。

这意味着：

- 页面不能只展示静态结果。
- 用户需要看到“AI 正在做什么”的过程透明度。
- 结果出现前，页面先有明确的进度反馈。

### Recommended Step Sequence

方案沟通页建议采用以下顺序：

1. 理解原始需求。
2. 提炼目标与限制条件。
3. 生成多个方案方向。
4. 用户选择其一。
5. AI 基于选择生成最终执行方案。
6. 用户确认后生成任务。

## Option Selection

用户需要看到多个明确的方案方向，而不是一开始只给一个答案。

每个方案卡片建议包含：

- 方案名称。
- 一句话定位。
- 适用场景或风格关键词。
- 关键信息标签，例如节奏、语气、镜头组织方式。
- 当前是否已选中。

交互上必须支持：

- 一次只确认一个主方向。
- 选中状态清晰可见。
- 最终确认方案在对话流中内联显示，而不是跳到独立侧栏。

## Final Plan Presentation

最终方案需要继续显示在对话流中，作为 AI 给出的结构化结果。

最终方案区建议包含：

- 标题。
- 风格方向。
- 时长或节奏设定。
- 核心段落拆分。
- 输出目标或落地说明。

这个区块是“会话中的确认结果”，不是另一个并排管理面板。

## Composer

底部输入区用于：

- 继续补充信息。
- 对方案提出修改。
- 触发下一轮方案细化。

在计划待确认阶段，需要提供清晰的主动作按钮，例如“确认方案并生成任务”。

## Page 3: Task Manager

## Purpose

任务页不是只读看板，而是一个可管理的任务列表界面。用户要能在这里筛选、查找、批量理解任务状态，并通过弹窗进入任务详情。

## Confirmed Layout

页面主体应以列表管理为核心：

- 顶部工具栏：搜索、筛选、视图范围、批量动作入口。
- 中间列表：任务主表格或强列表结构。
- 详情展示：使用弹窗，不使用常驻右侧详情栏。

## Task List Requirements

列表需要具备产品级管理感，至少覆盖：

- 任务标题与摘要。
- 当前状态。
- 进度。
- 所属队列、更新时间、负责人或来源信息。
- 快捷操作入口。

推荐支持的列表能力：

- 状态筛选。
- 搜索。
- 选择多条任务。
- 明确的选中态。
- 从列表直接进入详情弹窗。

## Task Detail Modal

任务详情必须使用弹窗承载。

弹窗内建议展示：

- 任务标题与状态。
- 总体进度。
- 当前阶段说明。
- 关键摘要指标。
- 事件时间线。
- 输出结果、异常或重试入口。

弹窗应服务于“查看和处理当前任务”，而不是把整个页面再复制一遍。

## Mapping to Existing Frontend

当前代码中，首页直接渲染 `AgentWorkspace`，并由以下组件组成：

- `AgentWorkspace`
- `AgentChat`
- `PlanPanel`
- `ProgressPanel`
- `ResultPanel`

后续实现建议按以下方式演进：

1. 首页从单一 Agent 工作台，调整为独立 Dashboard 页面。
2. 现有 Agent 会话能力迁移为 `/workspace` 主页面基础。
3. `PlanPanel`、`ProgressPanel`、`ResultPanel` 的职责重新拆分后，优先融入单栏方案沟通流，而不是继续保持右侧三块并列。
4. 新建任务管理页面承接任务列表和详情弹窗。

这意味着本轮更接近一次前端信息架构重组，而不是仅修改一套皮肤。

## Styling Strategy

实现阶段建议继续优先使用现有 CSS Modules 和全局变量体系，不强制在本轮全面切换到 Tailwind CSS。

原因：

- 当前代码已经基于 CSS Modules。
- 这次重点是信息架构和页面结构重组，不是样式方案迁移。
- 若同时做 Tailwind 全量迁移，会把风险从“页面重构”扩大到“样式体系重写”。

更合适的策略是：

1. 先把三个页面结构和交互落地。
2. 在变量和通用样式层统一新的产品化视觉。
3. 如果后续需要更高的组件复用或设计系统抽象，再评估 Tailwind 渐进引入。

## Data and State

本轮不要求改变核心后端接口契约，但前端页面需要按场景重组已有数据：

- Dashboard：聚合会话、任务、结果与趋势信息。
- Workspace：消费现有会话消息、状态、计划、事件。
- Tasks：消费任务执行状态、事件、结果摘要。

如果现有接口不足以支撑 Dashboard 或任务页摘要展示，可以在实现计划里补充最小必要的聚合接口，但不在本设计阶段强制定义新 API。

## Error Handling

- 方案沟通页要保留会话失效、发送失败、确认失败等错误反馈。
- 任务页要能清楚区分运行中、失败、完成三类状态。
- 任务详情弹窗里若存在错误，应明确显示失败原因与可执行动作。
- Dashboard 不应被单个任务异常打断，局部模块失败应局部提示。

## Responsive Behavior

### Tablet / Mobile

- 左侧 rail 可折叠或压缩为顶部导航。
- Dashboard 的双区布局收敛为单列堆叠。
- 方案沟通页保持单栏，不在移动端拆成多区。
- 任务页列表在窄屏下可转成卡片化列表，但详情仍然通过弹窗或全屏层展示。

## Verification

实现完成后至少需要验证：

1. 三个页面都可以稳定访问。
2. 首页数据区块在桌面和移动端都不出现拥挤或错位。
3. 方案沟通页中，AI 步骤遵循“先进度、后结果”的展示规则。
4. 用户原始 prompt 不被前端改写混入结构化标签。
5. 方案选项只能明确选择一个主方向，并能生成最终方案展示。
6. 任务页列表支持基本筛选、搜索和详情查看。
7. 任务详情以弹窗形式出现，而不是侧栏。
8. 当前会话恢复、运行中轮询、完成态结果展示不回退。
